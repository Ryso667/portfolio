Infrastructure Réseaux & Systèmes

Plateforme de Services Réseaux SMARTTECH

Infrastructure réseau complète et sécurisée pour PME, intégrant DNS centralisé, serveur de messagerie, accès distant multi-protocoles, partage de fichiers, intranet web et système de notifications automatiques par email.

Période

Février 2026

Type

Projet académique individuel

Contexte

Projet de fin de module Services Réseaux - L3 GLSI

Durée

3 semaines

Contexte & Objectifs

Conception et déploiement d'une infrastructure réseau complète pour la PME fictive SMARTTECH, permettant aux employés de travailler à distance en toute sécurité, de partager des fichiers, de communiquer par email interne et d'accéder à un intranet de gestion.

Problème résolu : Centraliser tous les services essentiels d'une entreprise sur un unique serveur Linux sécurisé, réduisant ainsi les coûts d'infrastructure tout en garantissant l'accès distant et la supervision automatique.

Public cible : PME de 10-50 employés nécessitant une infrastructure réseau professionnelle sans budget pour des solutions propriétaires coûteuses.

Mon Rôle & Responsabilités

  • Architecture réseau : Conception du schéma réseau complet avec DNS centralisé et segmentation logique des services.
  • Déploiement serveur Linux : Installation et configuration d'Ubuntu Server 22.04 LTS sur machine virtuelle (UTM sur Mac M2 Pro) avec IP statique et mode pont.
  • Service DNS (BIND9) : Configuration des zones directes et inverses pour le domaine smarttech.sn avec 10+ enregistrements (mail, ssh, rdp, vnc, intranet, etc.).
  • Accès distant sécurisé : Mise en place de SSH avec authentification par clé, RDP (xRDP), VNC (x11vnc) et noVNC pour accès navigateur.
  • Serveur de messagerie : Déploiement complet d'iRedMail (Postfix, Dovecot, Roundcube) avec création de comptes email et tests d'envoi/réception.
  • Partage de fichiers : Configuration Samba avec 3 niveaux d'accès (admin, users, public) et SFTP pour transferts sécurisés.
  • Application web intranet : Développement d'une application CRUD en PHP avec Nginx, MariaDB pour la gestion des employés, accessible en HTTPS.
  • Automatisation et monitoring : Création de 6 scripts Bash de surveillance (connexions SSH, services DOWN, tentatives échouées, commandes sudo, actions CRUD, accès Samba) avec notifications email automatiques via services systemd.
  • Sécurisation : Configuration firewall nftables avec règles précises, désactivation accès root SSH, chiffrement SSL/TLS pour tous les services web.
  • Documentation : Rédaction d'un rapport technique complet de 40+ pages avec captures d'écran, schémas réseau et section détaillée sur les difficultés rencontrées.

Apprentissages clés :

  • Maîtrise approfondie de l'administration système Linux (Ubuntu Server)
  • Configuration avancée de services réseau (DNS, SMTP, IMAP, VNC)
  • Scripting Bash pour automatisation et monitoring système
  • Gestion de firewall avec nftables (règles, ordre d'évaluation)
  • Architecture client-serveur et protocoles réseau (TCP/IP, SSH, RDP, VNC, SMB)
  • Diagnostic réseau avec outils Linux (ss, netstat, dig, tcpdump)
  • Déploiement et configuration Nginx avec virtual hosts multiples
  • Intégration PHP-FPM et MariaDB pour applications web
  • Sécurisation des communications (SSL/TLS, authentification par clé)
  • Documentation technique professionnelle avec schémas et captures

Défis techniques surmontés :

  • IP statique avec cloud-init : Configuration netplan pour éviter le changement d'IP à chaque redémarrage
  • SSH en IPv6 uniquement : Forcer l'écoute IPv4 avec AddressFamily inet dans sshd_config
  • Conflit UFW / nftables : Désactivation UFW et gestion exclusive via nftables avec ordre des règles correct
  • Coexistence Nginx : Configuration de virtual hosts pour iRedMail et intranet avec server_name spécifiques
  • PHP-FPM sur port TCP : Adaptation de la config Nginx pour fastcgi_pass 127.0.0.1:9999
  • VNC sans GUI : Utilisation de Xvfb pour créer un serveur X virtuel sur Ubuntu Server
  • Emails invisibles Roundcube : Compréhension du fonctionnement Maildir (new/ vs cur/)
